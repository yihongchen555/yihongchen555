Package Management CSharp Dotnet Edition

I was intrigued after taking a serious look at how .NET developers handles their dotnet packages

The main problem is that the .csproj file in one of module/microservice has references to other modules/microservices, so it is like nested dependencies on other directories, you have to find out the dependencies yourself by opening up other directories one by one (if you are not using Visual Studio)

For as long as I can remember, in other languages like Javascript, Python, PHP, Java and Golang, we package each microservice/module into a package and upload it to a package registry server. When we need a package we can add it into the package management file, e.g. package.json and then package-lock.json for Javascript

I am going to dive into C# .NET ecosystem to see if using a package registry is a common pattern

This is because from the CI/CD standpoint, to make the build pipeline fast enough, more configurable and easy to understand, this should be the default

By the way, the standards of all stuff should be set by DevOps/SRE, that includes everything in the CI/CD pipeline excluding writing the code

An example of the wrong way is how .NET developer using GUI tool to handle the package, see Referencing another project in .Net Core

I haven't seen any DevOps/SRE person handling the package management part using the IDE in my life yet, maybe one day
